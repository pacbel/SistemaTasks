1. INTRODUÇÃO

1.1 Propósito do Documento

Este documento apresenta a especificação técnica detalhada para o desenvolvimento de um novo sistema de gestão clínica para a Fundação Guimarães Rosa (FGR), que substituirá o atual sistema Coliseu. O documento inclui requisitos funcionais, não funcionais, arquitetura técnica, modelos de dados e diretrizes de implementação.

1.2 Escopo do Projeto

O projeto contempla o desenvolvimento de um sistema integrado para gestão completa da Clínica de Saúde da FGR, abrangendo funcionalidades como agendamento de consultas, gestão de prontuários, cadastro de profissionais e pacientes, faturamento de convênios, controle financeiro e relatórios gerenciais.

1.3 Definições, Acrônimos e Abreviações

  • REST: Representational State Transfer
  • SGBD: Sistema Gerenciador de Banco de Dados
  • UI/UX: User Interface/User Experience

2. DESCRIÇÃO GERAL DO SISTEMA

2.1 Perspectiva do Produto

O sistema proposto será uma aplicação web moderna, desenvolvida com tecnologias de ponta (NextJS, TypeScript, Prisma e MySQL), que substituirá integralmente o atual sistema Coliseu. Será projetado para oferecer uma experiência de usuário intuitiva e responsiva, mantendo todas as funcionalidades existentes e adicionando melhorias significativas.

2.2 Funções do Produto

O sistema de gestão clínica deverá:

  • Gerenciar o agendamento de consultas e procedimentos
  • Administrar cadastros de pacientes, médicos, funcionários e convênios
  • Processar faturamento para diversos convênios médicos
  • Gerenciar prontuários eletrônicos
  • Controlar aspectos financeiros (notas fiscais, pagamentos, repasses)
  • Gerar relatórios gerenciais e estatísticos
  • Controlar acesso com diferentes níveis de permissão
  • Automatizar alertas de vencimento de documentos/certificações
  • Otimizar processos administrativos

2.3 Características do Usuário

O sistema contará com os seguintes perfis de usuário:

  • Administrador: acesso completo ao sistema
  • Recepcionista: agendamento, cadastro de pacientes, atendimento presencial
  • Médico: acesso à agenda e prontuários dos seus pacientes
  • Financeiro: acesso aos módulos de faturamento, notas fiscais e repasses
  • Gestor: acesso a relatórios gerenciais e estatísticos

2.4 Restrições e Limitações

  • O sistema deve operar em ambiente web, com suporte a navegadores modernos
  • Deve implementar controles de acesso baseados em perfil
  • Deve garantir a confidencialidade dos dados médicos
  • Deve estar em conformidade com a LGPD
  • Deve permitir a migração completa dos dados do sistema Coliseu

2.5 Pressupostos e Dependências

  • Disponibilidade de infraestrutura de servidor adequada
  • Conectividade com internet estável na clínica
  • Acesso aos dados do sistema Coliseu para migração
  • Colaboração dos usuários-chave durante o desenvolvimento

3. REQUISITOS FUNCIONAIS

3.1 Gestão de Usuários e Acessos

RF-3.1.1 Cadastro e Gestão de Usuários

  • O sistema deve permitir o cadastro e manutenção de usuários (funcionários e médicos)
  • Deve registrar informações como nome, e-mail, telefone, cargo, setor e tipo de usuário
  • Deve possibilitar a definição de perfis de acesso específicos
  • Deve permitir a ativação/desativação de usuários

RF-3.1.2 Autenticação e Controle de Acesso

  • O sistema deve implementar autenticação segura via e-mail/senha
  • Deve suportar autenticação em dois fatores (opcional)
  • Deve implementar controle de acesso baseado em perfis
  • Deve registrar logs de acesso e operações críticas
  • Deve implementar timeout de sessão por inatividade
  • Deve permitir a recuperação de senha via e-mail

3.2 Cadastros Básicos

RF-3.2.1 Cadastro de Pacientes/Beneficiários

  • O sistema deve permitir o cadastro completo de pacientes
  • Deve registrar dados pessoais, documentos, contatos e convênios
  • Deve validar CPF para evitar duplicidade de cadastros
  • Deve permitir múltiplos convênios por paciente
  • Deve vincular números de carteirinhas aos respectivos convênios
  • Deve armazenar histórico de atendimentos
  • Deve possibilitar a importação de cadastros do sistema Coliseu

RF-3.2.2 Cadastro de Profissionais de Saúde

  • O sistema deve permitir o cadastro de médicos e outros profissionais
  • Deve registrar dados como CRM, RQE e especialidades
  • Deve armazenar documentos profissionais digitalizados
  • Deve registrar dados da empresa para emissão de notas fiscais
  • Deve possibilitar o acompanhamento de certificações e suas validades
  • Deve permitir a configuração de percentuais de repasse por convênio

RF-3.2.3 Cadastro de Convênios e Planos de Saúde

  • O sistema deve permitir o cadastro de operadoras de planos de saúde
  • Deve registrar dados específicos de cada convênio (IPSM, AMIL, COPASA, etc.)
  • Deve permitir a configuração de regras de faturamento por convênio
  • Deve registrar tabelas de preços e procedimentos
  • Deve configurar prazos de retorno específicos por convênio
  • Deve permitir a definição de formatos de carteirinhas por convênio

RF-3.2.4 Cadastro de Especialidades e Procedimentos

  • O sistema deve permitir o cadastro de especialidades médicas
  • Deve permitir o cadastro de procedimentos médicos
  • Deve vincular procedimentos a especialidades
  • Deve permitir a configuração de valores por procedimento/convênio
  • Deve possibilitar a importação de tabelas TUSS/AMB

3.3 Gestão de Agendas

RF-3.3.1 Configuração de Agendas

  • O sistema deve permitir a criação e configuração de agendas médicas
  • Deve possibilitar a definição de dias e horários de atendimento
  • Deve permitir a configuração de intervalos entre consultas
  • Deve suportar a abertura de agendas quinzenais ou mensais
  • Deve exibir visualmente no calendário as datas disponíveis

RF-3.3.2 Agendamento de Consultas

  • O sistema deve permitir o agendamento de consultas por especialidade ou médico
  • Deve exibir claramente os horários disponíveis
  • Deve verificar automaticamente a validade do convênio
  • Deve verificar o prazo de retorno do paciente conforme regras do convênio
  • Deve permitir a marcação de consultas recorrentes
  • Deve registrar o canal de agendamento (telefone, presencial, etc.)

RF-3.3.3 Gestão de Encaixes

  • O sistema deve permitir o encaixe de pacientes em horários já ocupados
  • Deve registrar a autorização do médico para o encaixe
  • Deve alertar sobre potenciais atrasos nos atendimentos

RF-3.3.4 Cancelamentos e Remarcações

  • O sistema deve permitir o cancelamento de consultas
  • Deve registrar o motivo do cancelamento
  • Deve permitir a remarcação rápida de consultas canceladas
  • Deve possibilitar o cancelamento de blocos de agenda (por período)
  • Deve notificar pacientes sobre cancelamentos

RF-3.3.5 Confirmação de Consultas

  • O sistema deve gerar automaticamente lista de consultas a confirmar
  • Deve registrar confirmações e ausências
  • Deve suportar envio automático de lembretes por WhatsApp
  • Deve personalizar mensagens de confirmação por especialidade

3.4 Atendimento

RF-3.4.1 Recepção e Check-in

  • O sistema deve permitir o check-in do paciente na recepção
  • Deve notificar o médico sobre a chegada do paciente
  • Deve registrar horário de chegada e tempo de espera
  • Deve alertar sobre documentação pendente ou irregular

RF-3.4.2 Validação de Convênios

  • O sistema deve validar a carteirinha do convênio no momento do atendimento
  • Deve verificar carências e coberturas
  • Deve solicitar autorização quando necessário
  • Deve registrar número de autorização para faturamento

RF-3.4.3 Prontuário Eletrônico

  • O sistema deve disponibilizar prontuário eletrônico por paciente
  • Deve permitir o registro de anamnese, diagnósticos e condutas
  • Deve armazenar histórico completo de atendimentos
  • Deve suportar anexos de exames e documentos
  • Deve implementar controles rigorosos de privacidade

RF-3.4.4 Emissão de Documentos

  • O sistema deve gerar declarações de comparecimento
  • Deve gerar receitas médicas
  • Deve gerar solicitações de exames
  • Deve gerar atestados médicos
  • Deve permitir a personalização de documentos por médico

3.5 Faturamento

RF-3.5.1 Processamento de Guias

  • O sistema deve gerar guias de atendimento conforme padrões TISS
  • Deve gerenciar guias por status (aberta, autorizada, faturada)
  • Deve permitir a inclusão manual de procedimentos quando necessário
  • Deve validar procedimentos conforme tabelas dos convênios

RF-3.5.2 Faturamento de Convênios

  • O sistema deve gerar lotes de faturamento por convênio
  • Deve suportar periodicidades específicas por convênio
  • Deve adaptar-se aos requisitos de cada convênio (IPSM, AMIL, CEMIG, etc.)
  • Deve permitir faturamento manual e automatizado
  • Deve gerar protocolos e relatórios para envio físico quando necessário
  • Deve controlar prazos de envio de faturamento

RF-3.5.3 Controle de Recebimentos

  • O sistema deve registrar recebimentos por convênio
  • Deve permitir a conciliação de pagamentos
  • Deve identificar glosas e permitir recursos
  • Deve gerar relatórios de valores a receber

RF-3.5.4 Repasse aos Profissionais

  • O sistema deve calcular automaticamente repasses aos médicos
  • Deve considerar percentuais configurados por convênio/procedimento
  • Deve gerar extratos detalhados para os profissionais
  • Deve gerar cartas de solicitação de notas fiscais
  • Deve controlar o recebimento de notas fiscais
  • Deve gerar relatórios para autorização de pagamentos

3.6 Gestão Financeira

RF-3.6.1 Emissão de Notas Fiscais

  • O sistema deve integrar-se com sistema de emissão de NFS-e
  • Deve gerar notas fiscais para atendimentos particulares e convênios
  • Deve registrar e controlar notas emitidas
  • Deve gerar relatórios de faturamento por período
  • Deve preparar dados para DMED anual

RF-3.6.2 Controle de Pagamentos

  • O sistema deve registrar pagamentos por diferentes meios (dinheiro, cartão, PIX)
  • Deve gerar relatórios de recebimentos por meio de pagamento
  • Deve permitir o controle de descontos em folha
  • Deve gerar relatórios para envio ao financeiro

RF-3.6.3 Relatórios Financeiros

  • O sistema deve gerar relatórios de faturamento por período
  • Deve gerar relatórios por convênio
  • Deve gerar relatórios por profissional
  • Deve gerar relatórios de inadimplência
  • Deve permitir a exportação em formatos editáveis (CSV, Excel)

3.7 Gestão Administrativa

RF-3.7.1 Controle de Documentação

  • O sistema deve gerenciar documentos da clínica (alvarás, certificações)
  • Deve alertar sobre vencimentos próximos
  • Deve armazenar cópias digitais dos documentos
  • Deve registrar histórico de renovações

RF-3.7.2 Gestão de Estoque

  • O sistema deve controlar estoque de materiais médicos e de escritório
  • Deve alertar sobre níveis baixos de estoque
  • Deve registrar solicitações de compra
  • Deve registrar recebimentos de material

RF-3.7.3 Comunicações

  • O sistema deve permitir o envio de comunicados aos profissionais
  • Deve gerenciar modelos de mensagens
  • Deve registrar histórico de comunicações

3.8 Relatórios e Dashboards

RF-3.8.1 Relatórios Operacionais

  • O sistema deve gerar relatórios de agendamentos por período
  • Deve gerar relatórios de atendimentos realizados
  • Deve gerar relatórios de faltas e cancelamentos
  • Deve gerar relatórios de tempos de espera

RF-3.8.2 Relatórios Gerenciais

  • O sistema deve gerar relatórios de produtividade por profissional
  • Deve gerar relatórios de faturamento por especialidade
  • Deve gerar relatórios comparativos entre períodos
  • Deve permitir a criação de relatórios personalizados

RF-3.8.3 Dashboards

  • O sistema deve apresentar dashboards interativos
  • Deve exibir indicadores-chave de desempenho (KPIs)
  • Deve permitir a personalização de visualizações
  • Deve atualizar dados em tempo real

4. REQUISITOS NÃO FUNCIONAIS

4.1 Requisitos de Usabilidade

RNF-4.1.1 Interface de Usuário

  • A interface deve ser intuitiva e responsiva, adaptando-se a diferentes dispositivos
  • Deve seguir princípios de design centrado no usuário
  • Deve apresentar feedback visual para ações importantes
  • Deve utilizar componentes consistentes em toda a aplicação

RNF-4.1.2 Experiência do Usuário

  • O sistema deve permitir navegação eficiente com poucos cliques
  • Deve implementar atalhos de teclado para operações frequentes
  • Deve exibir mensagens de erro claras e orientativas
  • Deve permitir a personalização de preferências de visualização

4.2 Requisitos de Desempenho

RNF-4.2.1 Tempo de Resposta

  • O sistema deve carregar páginas principais em menos de 2 segundos
  • Operações de busca devem retornar resultados em menos de 3 segundos
  • O agendamento completo deve ser realizado em menos de 30 segundos
  • Relatórios simples devem ser gerados em menos de 5 segundos

RNF-4.2.2 Escalabilidade

  • O sistema deve suportar até 50 usuários simultâneos na fase inicial
  • Deve ser escalável para suportar crescimento de 20% ao ano no volume de dados
  • Deve manter desempenho estável durante o horário de pico

4.3 Requisitos de Segurança

RNF-4.3.1 Autenticação e Autorização

  • O sistema deve implementar autenticação segura com senhas fortes
  • Deve utilizar JWT para gerenciamento de sessões
  • Deve implementar controle de acesso baseado em perfis (RBAC)
  • Deve registrar tentativas de acesso não autorizado

RNF-4.3.2 Confidencialidade e Privacidade

  • O sistema deve criptografar dados sensíveis no banco de dados
  • Deve implementar criptografia em trânsito (HTTPS)
  • Deve estar em conformidade com a LGPD
  • Deve implementar políticas de privacidade para dados médicos

RNF-4.3.3 Auditoria

  • O sistema deve manter logs de todas as ações críticas
  • Deve registrar quem fez o quê, quando e a partir de onde
  • Deve permitir a rastreabilidade de alterações em dados sensíveis
  • Deve implementar mecanismos contra adulteração de logs

4.4 Requisitos de Confiabilidade

RNF-4.4.1 Disponibilidade

  • O sistema deve estar disponível 99,5% do tempo durante horário comercial
  • Deve implementar mecanismos de recuperação após falhas
  • Deve permitir o agendamento de manutenções programadas

RNF-4.4.2 Backup e Recuperação

  • O sistema deve realizar backups diários automaticamente
  • Deve permitir a restauração de dados para um ponto específico no tempo
  • Deve manter histórico de backups por pelo menos 90 dias
  • Deve testar periodicamente a integridade dos backups

4.5 Requisitos de Manutenibilidade

RNF-4.5.1 Modularidade

  • O sistema deve ser desenvolvido com arquitetura modular
  • Deve permitir a atualização de componentes específicos sem afetar todo o sistema
  • Deve utilizar padrões de design que facilitem a manutenção

RNF-4.5.2 Testabilidade

  • O código deve ter cobertura de testes unitários mínima de 80%
  • Deve implementar testes de integração para fluxos críticos
  • Deve suportar testes automatizados de UI

4.6 Requisitos de Portabilidade

RNF-4.6.1 Compatibilidade

  • O sistema deve ser compatível com navegadores modernos (Chrome, Firefox, Edge, Safari)
  • Deve funcionar em dispositivos diferentes (desktop, tablet, smartphones)
  • Deve suportar sistemas operacionais diversos (Windows, MacOS, Linux)

RNF-4.6.2 Interoperabilidade

  • O sistema deve oferecer APIs RESTful para integração com outros sistemas
  • Deve suportar a importação/exportação de dados em formatos padrão
  • Deve permitir a integração com sistemas de prontuário eletrônico

4.7 Requisitos de Implantação

RNF-4.7.1 Instalação

  • O sistema deve ser implantado em ambiente de nuvem ou on-premises
  • Deve permitir a instalação com configuração mínima de parâmetros
  • Deve fornecer scripts de implantação automatizados

RNF-4.7.2 Atualizações

  • O sistema deve suportar atualizações sem perda de dados
  • Deve permitir rollback em caso de problemas
  • Deve implementar versionamento semântico

5. ARQUITETURA TÉCNICA

5.1 Visão Geral da Arquitetura

O sistema será desenvolvido como uma aplicação web moderna, utilizando arquitetura de cliente-servidor com frontend web responsivo. A arquitetura seguirá os princípios de Clean Architecture, separando claramente as responsabilidades em camadas.

5.2 Tecnologias Utilizadas

5.2.1 Frontend

  • Framework: Next.js 14.0+
  • Linguagem: TypeScript 5.0+
  • Biblioteca de UI: React 18+
  • Gerenciamento de Estado: React Context API e SWR
  • Estilização: Tailwind CSS + Styled Components
  • Validação de Formulários: React Hook Form + Zod
  • Cliente HTTP: Axios
  • Testes: Jest + React Testing Library

5.2.2 Backend

  • Framework: Next.js API Routes
  • Linguagem: TypeScript 5.0+
  • ORM: Prisma
  • SGBD: MySQL 8.0+
  • Autenticação: NextAuth.js + JWT
  • Validação: Zod
  • Logging: Winston
  • Testes: Jest + Supertest

5.3 Organização do Código

5.3.1 Estrutura de Diretórios

/ ├─ prisma/ # Configuração do Prisma e modelos ├─ public/ # Arquivos públicos estáticos ├─ src/ │ ├─ app/ # Componentes de página do Next.js │ ├─ components/ # Componentes React reutilizáveis │ │ ├─ ui/ # Componentes de UI básicos │ │ ├─ forms/ # Componentes de formulário │ │ ├─ layouts/ # Componentes de layout │ │ └─ [module]/ # Componentes específicos de módulos │ ├─ contexts/ # Contextos React para estado global │ ├─ hooks/ # Custom hooks │ ├─ lib/ # Utilitários e helpers │ ├─ models/ # Tipos e interfaces │ ├─ pages/ # Páginas do Next.js │ │ └─ api/ # Rotas de API │ ├─ server/ # Código específico do servidor │ │ ├─ auth/ # Lógica de autenticação │ │ ├─ db/ # Lógica de acesso a dados │ │ └─ services/ # Serviços do backend │ ├─ styles/ # Estilos globais │ └─ utils/ # Funções utilitárias ├─ tests/ # Testes └─ .env # Variáveis de ambiente

5.4 Padrões de Design

  • Repository Pattern: para acesso a dados
  • Service Layer: para lógica de negócios
  • Component-Based Architecture: para o frontend
  • Hooks Pattern: para lógica reutilizável no React
  • Clean Architecture: separação de responsabilidades

5.5 Fluxo de Dados

  1. Cliente para Servidor: Requisições HTTP via Axios para as API Routes
  2. Processamento no Servidor: Validação → Autorização → Serviço → Repositório → Banco de Dados
  3. Servidor para Cliente: Respostas JSON com dados estruturados
  4. Renderização no Cliente: Atualização do estado React e re-renderização de componentes

5.6 Estratégias de Deploy

  • Ambiente de Desenvolvimento: Deploy local para testes
  • Ambiente de Homologação: Deploy em servidor de staging
  • Ambiente de Produção: Deploy em servidor de produção
  • CI/CD: GitHub Actions para automação de testes e deploy

6. MODELO DE DADOS

6.1 Diagrama Entidade-Relacionamento

O sistema utilizará um banco de dados relacional MySQL, com o seguinte esquema conceitual (principais entidades):

  • Usuários (users)
  • Pacientes (patients)
  • Profissionais (professionals)
  • Empresas (companies)
  • Convênios (health_plans)
  • Especialidades (specialties)
  • Procedimentos (procedures)
  • Agendas (schedules)
  • Consultas (appointments)
  • Atendimentos (medical_records)
  • Guias (service_orders)
  • Faturamentos (billings)
  • Pagamentos (payments)
  • Repasses (professional_payments)
  • Documentos (documents)

6.2 Principais Entidades

6.2.1 Usuário (User)

  • id: UUID (PK)
  • username: String (Unique)
  • email: String (Unique)
  • password_hash: String
  • role: Enum (admin, doctor, receptionist, etc.)
  • is_active: Boolean
  • last_login: DateTime
  • created_at: DateTime
  • updated_at: DateTime

6.2.2 Paciente (Patient)

  • id: UUID (PK)
  • name: String
  • cpf: String (Unique)
  • birth_date: Date
  • gender: Enum
  • address: Embedded Object
  • contact: Embedded Object
  • health_plans: Relation (PatientHealthPlan)
  • appointments: Relation (Appointment)
  • medical_records: Relation (MedicalRecord)
  • created_at: DateTime
  • updated_at: DateTime

6.2.3 Profissional (Professional)

  • id: UUID (PK)
  • user_id: UUID (FK to User)
  • name: String
  • crm: String
  • rqe: String
  • specialties: Relation (ProfessionalSpecialty)
  • company_id: UUID (FK to Company)
  • schedules: Relation (Schedule)
  • appointments: Relation (Appointment)
  • commission_rates: Relation (CommissionRate)
  • created_at: DateTime
  • updated_at: DateTime

6.2.4 Convênio (HealthPlan)

  • id: UUID (PK)
  • name: String
  • code: String
  • billing_day: Integer
  • return_period: Integer
  • card_format: String
  • payment_type: Enum
  • procedures: Relation (HealthPlanProcedure)
  • patients: Relation (PatientHealthPlan)
  • created_at: DateTime
  • updated_at: DateTime

6.2.5 Consulta (Appointment)

  • id: UUID (PK)
  • patient_id: UUID (FK to Patient)
  • professional_id: UUID (FK to Professional)
  • health_plan_id: UUID (FK to HealthPlan)
  • specialty_id: UUID (FK to Specialty)
  • date_time: DateTime
  • duration: Integer
  • status: Enum (scheduled, confirmed, completed, cancelled)
  • cancellation_reason: String
  • scheduling_channel: Enum
  • is_return: Boolean
  • service_order_id: UUID (FK to ServiceOrder)
  • created_at: DateTime
  • updated_at: DateTime

6.2.6 Faturamento (Billing)

  • id: UUID (PK)
  • health_plan_id: UUID (FK to HealthPlan)
  • start_date: Date
  • end_date: Date
  • status: Enum (pending, processing, completed)
  • total_value: Decimal
  • protocol_number: String
  • service_orders: Relation (ServiceOrder)
  • created_at: DateTime
  • updated_at: DateTime

6.2.7 Ordem de Serviço (ServiceOrder)

  • id: UUID (PK)
  • appointment_id: UUID (FK to Appointment)
  • patient_id: UUID (FK to Patient)
  • professional_id: UUID (FK to Professional)
  • health_plan_id: UUID (FK to HealthPlan)
  • procedure_id: UUID (FK to Procedure)
  • authorization_number: String
  • status: Enum (pending, authorized, denied, billed)
  • value: Decimal
  • billing_id: UUID (FK to Billing)
  • created_at: DateTime
  • updated_at: DateTime
